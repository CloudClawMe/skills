# Publish File Skill

## Коротко

Копирует файл в `/var/www/html/static/` внутри контейнера пользователя и возвращает HTTPS-ссылку на его default domain. Под капотом — преднастроенный Nginx. Публикация дешёвая (одна shell-команда + один API-запрос), файл **публичный** по прямой ссылке и живёт **только пока жив контейнер**. Никакого отдельного storage backend нет.

Примеры того, как выглядит результат:

- «закинь этот PDF, дай линк» → `https://alice.myproj.me/static/a1b2c3d4e5f6/report.pdf`
- «выложи картинку» → `https://alice.myproj.me/static/7890abcdef12/photo.jpg`

## Что уже настроено в контейнере

Каждый user container CloudClaw приходит преднастроенным:

- Nginx слушает `:80`, единый `/etc/nginx/nginx.conf` содержит блок `location /static/`, обслуживающий директорию `/var/www/html/static/` с `autoindex off` и `expires 1d`;
- директория `/var/www/html/static/` существует и writable пользователем (ownership выставлен при инициализации контейнера);
- default domain вида `<name>.myproj.me` выдан и привязан к контейнеру — резолвится через внутренний shared service `domain-router` (internal DNS `http://domain-router`).

Скилл **полагается** на этот контракт и сам его не чинит. Если что-то сломано — см. раздел **Troubleshooting**.

## Публикация файла

### Шаг 1. Policy warning и подтверждение

Перед каждым upload'ом покажи пользователю предупреждение из `references/policy.md` и дождись явного согласия. Без подтверждения дальше не иди. Если пользователь отказался — остановись и скажи, что файл не опубликован.

Policy-блок — не формальность, а guardrail от abuse. Не пропускай.

### Шаг 2. Узнать default domain

Default domain запрашивается у shared service `domain-router` по внутреннему URL `http://domain-router`:

```bash
default_domain=$(curl -s "http://domain-router/api/domains" \
  | jq -r '.domains[] | select(.kind=="default") | .hostname')
```

Ответ:

```json
{
  "domains": [
    { "id": "...", "hostname": "alice.myproj.me", "kind": "default", "is_active": true, "created_at": "..." }
  ]
}
```

Default — тот, у которого `kind == "default"`. Если запрос упал — не DDoS'ь сервис, сообщи пользователю, что не смог получить домен, и попроси попробовать чуть позже.

### Шаг 3. Сгенерировать короткий `file_id`

`file_id` — имя подпапки в `/var/www/html/static/`. Используй случайные 12 hex-символов:

```bash
file_id=$(openssl rand -hex 6)
```

Используй подпапку, а не плоское имя `<id>-<filename>`, потому что:

- `rm -rf /var/www/html/static/<file_id>/` — атомарный delete без парсинга;
- оригинальное имя файла и расширение сохраняются в URL в чистом виде;
- случайные `ls` в директорию не выдают все чужие ID сразу (при `autoindex off` это и так защищено, но defence in depth не мешает).

### Шаг 4. Положить файл

```bash
mkdir -p "/var/www/html/static/${file_id}"
cp "<source_path>" "/var/www/html/static/${file_id}/<filename>"
```

`<filename>` — имя, которое прислал пользователь (или которое ты сам выбрал, если имени нет). Пробелы и спецсимволы лучше заменить на `_`, чтобы URL был чистым, но это необязательно — Nginx отдаст файл с любым именем.

### Шаг 5. Собрать URL и вернуть результат

```
https://<default_domain>/static/<file_id>/<filename>
```

Отдай пользователю:

- публичную ссылку;
- `file_id` (понадобится для удаления);
- размер файла (`stat -c %s <path>`);
- если подписка Trial — напоминание из раздела **Trial caveat** ниже.

## Удаление файла

Пользователь сообщает `file_id`:

```bash
rm -rf "/var/www/html/static/${file_id}/"
```

Ownership проверять не надо — пользователь оперирует в собственном контейнере, всё в `/var/www/html/static/` по определению его. Если подпапки нет — скажи об этом спокойно, не падай.

## Список файлов

Чтобы показать пользователю, что он уже опубликовал:

```bash
ls -la /var/www/html/static/
```

Для каждой подпапки собери:

- `file_id` (имя подпапки);
- `filename` (содержимое подпапки — обычно один файл);
- размер (`stat -c %s`);
- URL (собирается так же, как в шаге 5 публикации).

Отдай таблицей. Если подпапок нет — скажи «опубликованных файлов пока нет».

## Trial caveat

Если у пользователя подписка Trial, добавь к результату upload'а такую пометку:

> На Trial контейнер засыпает после **30 минут** неактивности. Пока контейнер спит, ссылка показывает страницу-заглушку «инстанс спит» вместо файла. HTTP-запросы к домену контейнер **не будят** — запустить его заново можно только через `@CloudClawBot`. Сам файл физически не удаляется при засыпании, так что после запуска контейнера ссылка снова отдаёт файл.
>
> Для стабильного шеринга рекомендуем тариф Base или выше.

## Жизненный цикл файлов

Файлы живут внутри контейнера:

- пока подписка **активна** — файлы доступны и хранятся на диске контейнера;
- когда подписка **заканчивается** — начинается 30-дневный grace period (контейнер и его данные сохраняются);
- по истечении grace period контейнер удаляется целиком, и с ним вся директория `/var/www/html/static/`. Отдельного cleanup-job для файлов нет.

Upgrade или downgrade тарифа файлы **не отзывает**. Если после даунгрейда диск тесноват — это уже задача пользователя разгрести.

## Лимиты и биллинг

- Отдельного программного лимита на размер одного файла нет. Реальный потолок — свободное место на диске контейнера, выделенное тарифом.
- Лимита на количество файлов нет.
- Биллинг в tool calls: upload / delete / list — по 1 tool call за операцию.
- Отдельного MB-биллинга нет: disk quota тарифа сам по себе работает как естественный guardrail. Если пользователь хочет хранить много тяжёлых файлов — естественный ответ апгрейд тарифа с бо́льшим диском.

## Policy

Запрещено публиковать: нелегальный контент, пиратское ПО / торренты / нелегальные дистрибуции, malware, порнографию, любой контент, нарушающий применимое законодательство.

Полный текст warning'а — в `references/policy.md`. Автоматической модерации в v1 нет: enforcement ручной (поддержка может зайти в контейнер и удалить файл; при повторных нарушениях — блокировка аккаунта). Policy warning в шаге 1 — обязательная часть flow.

## Troubleshooting

| Симптом | Причина и что делать |
|---|---|
| URL отдаёт `404 Not Found` | Файл не скопировался, опечатка в `<filename>` в URL, или пользователь удалил `location /static/` из своего `nginx.conf`. Проверь: `ls /var/www/html/static/<file_id>/`. Если `location /static/` удалён — восстанови через соседний скилл `site-publishing` (у него есть recovery-конфиг). |
| URL отдаёт страницу «инстанс спит» | Контейнер остановлен (обычно Trial после 30 минут неактивности). HTTP-запросы его не будят — запустить заново можно только через `@CloudClawBot`. Для стабильной работы — платный тариф. |
| `mkdir` / `cp` — `Permission denied` | Ownership `/var/www/html/static/` сбит. Исправь: `sudo chown -R $USER:$USER /var/www/html/static/`. |
| `curl http://domain-router/api/domains` упал | Shared-service временно недоступен. Не DDoS'ь; сообщи пользователю, что сейчас публикацию сделать нельзя, и предложи попробовать позже. |
| `nginx -t` ругается после ручных правок | Пользователь сломал конфиг. Это не к этому скиллу — отправь в соседний скилл `site-publishing` / recovery. |

## Границы скилла

Чего этот скилл **не** делает (если пользователь просит — скажи честно и предложи альтернативу):

- **Не делает приватный / авторизованный шеринг.** Всё, что выкладывается, доступно по прямой HTTPS-ссылке любому. Если нужен auth — это отдельная задача, в этот скилл не входит.
- **Не деплоит сайты и приложения.** Для статики, SPA, Docker-приложений за Nginx — соседний скилл `site-publishing` (`../site-publishing/SKILL.md`).
- **Не делает signed URLs и не ограничивает TTL ссылок.** Пока файл лежит в `/var/www/html/static/<file_id>/` — ссылка работает. TTL — это ответственность пользователя (удалить через delete flow).
- **Не проксирует через отдельный storage backend.** Файл физически лежит на диске контейнера, а не в S3 / object store.
- **Не обслуживает потоки (stream) и не делает transcoding.** Любой файл отдаётся как есть.
