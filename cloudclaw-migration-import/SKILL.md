# Migration Skill

Ты — ассистент нового CloudClaw инстанса. Пользователь хочет перенести свои данные из старого OpenClaw инстанса. Веди его по шагам из этого документа.

## Ключевой принцип

Миграция — **ручная пересылка dump-файла пользователем**. Пользователь видит файл, сам его пересылает из старого чата в новый, сам читает отчёт об импорте. Это важно, потому что dump содержит всю память пользователя, и управление процессом должно оставаться у него.

## Пошаговый flow

### Шаг 1. Объясни, как будет проходить миграция

Отправь пользователю этот текст (можно слегка адаптировать под стиль диалога, но суть и пункты сохрани):

```
Помогу перенести данные из старого инстанса.

Как это работает:
1. Я дам тебе готовый текст — перешли его в чат со старым инстансом.
2. Ассистент старого инстанса соберёт архив с твоей памятью, историями и скиллами.
3. Ты пришлёшь этот архив мне, сюда.
4. Я распакую всё в твой новый инстанс и покажу, что получилось.

Архив идёт напрямую через Telegram от тебя к тебе. Конфиг старого инстанса мы не переносим: он полностью управляется CloudClaw.

Готов?
```

Дождись подтверждения, что пользователь хочет продолжить.

### Шаг 2. Выдай текст для пересылки в старый инстанс

Отправь пользователю ровно этот блок (в виде отдельного сообщения, чтобы ему было удобно скопировать целиком):

```
Мне нужен dump моей памяти для миграции в новый инстанс CloudClaw.

Скачай и выполни инструкцию отсюда:
https://raw.githubusercontent.com/CloudClawMe/skills/main/cloudclaw-migration-import/old-instance-export/SKILL.md

Собери архив с моими .md-файлами, историями, пользовательскими скиллами и локальными БД. Конфиг инстанса НЕ включай. Про перенос моих личных секретов/API-ключей — спроси меня перед сборкой.

Когда будет готово — пришли мне файл в этот чат.
```

Затем добавь короткое сообщение «жду архив»:

```
Отлично. Когда ассистент старого инстанса пришлёт тебе файл — перешли его сюда. Я дождусь.

Если архив получится больше 2 GB, ассистент старого инстанса вместо файла пришлёт инструкцию по S3-хранилищу. Перешли её сюда — я помогу разобраться.
```

### Шаг 3. Прими dump и выполни silent import

Пользователь пересылает тебе один из двух входов:

**A) Архив документом в чат** (`.tar.gz` или `.zip`, ≤2 GB).

**B) Ссылку на S3** (если архив был > 2 GB).
Проверь ссылку: она должна быть либо presigned URL, либо содержать достаточно длинный случайный UID в пути (URL не должен выглядеть предсказуемо угадываемым). Скачай файл по ссылке во временный путь инстанса.

Если пользователь прислал ссылку на Google Drive, Dropbox, Яндекс.Диск (не S3-endpoint) или другое хранилище без прямого скачивания — скажи честно, что не сможешь скачать, и попроси переложить архив в S3-совместимое хранилище. См. шаблон ответа в разделе **S3 fallback** ниже.

Получив архив (из A или B), выполни **silent import**:

1. Распакуй архив (`.tar.gz` или `.zip`) в временный каталог.
2. Обходи содержимое и копируй файлы в пользовательский файлспейс нового инстанса, сохраняя относительные пути.
3. При конфликте имён применяй rename-policy (см. ниже).
4. Если отдельный файл не удаётся распаковать или скопировать — **пропусти его** и зафиксируй причину. Импорт как процесс не прерывай.
5. После обхода удали временные файлы (распакованные и сам архив).
6. Если файл пришёл через S3 — напомни пользователю удалить его из бакета (см. ниже).
7. Выдай финальный отчёт (см. раздел **Отчёт**).

Silent import выбран специально, чтобы миграция не ломалась из-за того, что dump оказался чуть-чуть другой формы, чем ожидалось. Что смогли — импортировали, что не смогли — честно сказали. Пользователь сам разберётся.

## Что попадает в файлспейс

Архив представляет собой срез пользовательской директории старого инстанса. Типичное содержимое:

- `.md`-файлы пользовательской памяти и заметок;
- локальные БД пользователя (`.db`, `.sqlite`, `.sqlite3`);
- папки пользовательских скиллов со всем содержимым (`SKILL.md` + артефакты);
- истории чата в JSONL;
- опционально — пользовательские секреты/API-ключи, если пользователь **явно разрешил** их перенос в шаге на стороне старого инстанса.

Конфиг и системные файлы OpenClaw в архив не попадают — это гарантируется companion-инструкцией на стороне старого инстанса. Никакого обязательного `manifest.json` или проверки версий в dump'е не требуется — не ищи его и не блокируй импорт на его отсутствии.

## Conflict policy (rename-keep-both)

Если в файлспейсе нового инстанса уже существует файл с тем же относительным путём:

- **Существующий файл остаётся нетронутым.**
- Импортируемый файл сохрани рядом, с суффиксом по шаблону:
  `<basename>.imported-<YYYY-MM-DD>.<ext>`
  Примеры:
  - `notes.md` → `notes.imported-2026-04-18.md`
  - `projects/todo.md` → `projects/todo.imported-2026-04-18.md`
- Каждую такую пару зафиксируй в отчёте (раздел «Переименовано»).

Rename вместо overwrite выбран потому, что пользователь мог уже что-то наработать в новом инстансе — перезапись молча съела бы эту работу. Оставляем два файла и даём пользователю решить вручную.

## S3 fallback (только если архив > 2 GB)

Если старый инстанс прислал пользователю не файл, а инструкцию про S3 — или пользователь сам говорит, что архив слишком большой для Telegram — отправь ему этот текст:

```
Архив слишком большой для Telegram (>2 GB). Нужно временное хранилище.

Что сделать:
1. Возьми любого S3-совместимого провайдера (Cloudflare R2, AWS S3, Yandex Cloud Storage).
2. Создай бакет, залей туда архив.
3. Получи ссылку на файл — либо presigned URL с TTL на пару часов, либо URL со случайным UID в пути (чтобы его нельзя было угадать).
4. Пришли ссылку мне сюда.

Google Drive, Dropbox и Яндекс.Диск без S3-endpoint'а не подойдут — я не смогу скачать файл по их ссылкам.
```

Автоматической генерации upload-ссылки на нашей стороне нет — пользователь сам готовит бакет и ссылку.

После успешного импорта из S3 обязательно отправь напоминание:

```
Миграция прошла успешно. Не забудь удалить архив из своего S3-бакета — он больше не нужен, а хранить dump с твоими данными в облаке без необходимости не стоит.
```

## Отчёт

После завершения импорта отправь пользователю отчёт в markdown. Используй эту структуру:

```markdown
**Миграция завершена.**

Импортировано: N файлов.
Переименовано (конфликты): M файлов.
Пропущено (ошибки): K файлов.

### Импортировано
- memory/goals.md
- memory/projects.md
- skills/my-custom-skill/SKILL.md
- history/2025-12.jsonl
...

### Переименовано
- notes.md → notes.imported-2026-04-18.md
- projects/todo.md → projects/todo.imported-2026-04-18.md

### Пропущено
- cache/tmp.bin — не удалось распаковать (битый файл)
- skills/broken-skill/index.bin — неожиданный формат
```

Пустую секцию можно опустить или явно написать «нет». Перечисляй реальные пути файлов, без выдуманных. Если список слишком длинный (>30 строк) — покажи первые 10 + количество остальных и предложи прислать полный список по запросу.

## Приватность

В любом разговоре про миграцию держи эти факты наготове — они важны и пользователь может спросить:

- Dump в default-сценарии живёт только в двух Telegram-чатах пользователя: старого и нового инстанса.
- Содержимое dump'а не логируется. В логах остаются только метаданные — имена, размеры, статусы импорта.
- Конфиг старого инстанса не переносится никогда.
- В S3-fallback ответственность за приватность ссылки и за удаление файла после импорта лежит на пользователе.

## Границы skill'а

Чего этот skill **не** делает (если пользователь просит — скажи честно и предложи альтернативу):

- Не экспортирует данные ИЗ текущего инстанса наружу.
- Не делает peer-to-peer авто-миграцию между двумя CloudClaw-инстансами — пользователь всегда вручную пересылает архив.
- Не валидирует dump до импорта и не блокирует импорт на «несовместимых версиях». Принцип — попытаться, отчитаться.
- Не перезаписывает существующие файлы пользователя. Только rename-keep-both.
