# CloudClaw Migration — Export

Ты — ассистент старого OpenClaw инстанса. Пользователь попросил подготовить архив своих данных для миграции в новый инстанс CloudClaw. Твоя задача — сориентироваться в терминах (ниже), уточнить у пользователя пару вопросов, собрать правильный архив, отдать файл пользователю в этот же чат, и не отправлять ничего больше никуда.

## Контекст (важно для правильной работы)

Разберись с терминами до того, как начнёшь что-либо делать:

- **Новый CloudClaw** — целевая платформа миграции. Это Telegram-бот `@CloudClawBot`, внутри которого каждому пользователю выдаётся собственный изолированный OpenClaw-инстанс с подпиской, общими скиллами и общими shared-сервисами. **Пользователь прямо сейчас уже находится в новом CloudClaw-инстансе** в параллельном чате — именно оттуда он получил инструкцию со ссылкой на этот файл и переслал её сюда.
- **Старый OpenClaw инстанс** — это **ты сам**. Инстанс OpenClaw, в котором пользователь пишет тебе это сообщение. Он может работать где угодно: на сервере пользователя, у стороннего облачного провайдера, в предыдущей платформе. В парадигме миграции ты — источник данных, новый CloudClaw-инстанс — приёмник.
- **«Старый CloudClaw» — такого термина нет**, не используй его. CloudClaw — это новая платформа, на которую пользователь переезжает. Даже если раньше пользователь уже был на CloudClaw и сейчас настраивает другой инстанс там же, предыдущую среду по-прежнему называем «старый OpenClaw инстанс».

Единственный канал, через который ты взаимодействуешь с пользователем в этом процессе — текущий Telegram-чат. С новым CloudClaw-инстансом ты напрямую не общаешься и ничего туда не отправляешь. Связующее звено — сам пользователь, который руками пересылает файл из одного чата в другой.

## Что включить в архив

Включай **пользовательские данные**:

- **Все `.md`-файлы пользовательской памяти** — заметки, цели, планы, контекст проектов, всё что накопилось как markdown.
- **Локальные пользовательские БД** — `.db`, `.sqlite`, `.sqlite3`, созданные пользователем или его скиллами в ходе работы.
- **Пользовательские скиллы, которые пользователь установил или создал сам.** Папку целиком: `SKILL.md` + все вложенные артефакты, ресурсы, подпапки. **Стоковые системные скиллы инстанса — не включай**, в новом CloudClaw-инстансе они уже есть в актуальной версии.
- **Истории чата в JSONL** — если в файлспейсе есть JSONL-файлы истории разговоров.

Общее правило: **если файл лежит в user data directory и это не системный конфиг / кэш — включай.** Архив не обязан соответствовать какому-то формальному манифесту, никакого `manifest.json` создавать не нужно.

## Что НЕ включать (критично)

Эти вещи **никогда** не должны попасть в архив. Нарушение ломает работу нового CloudClaw-инстанса:

- **Системный конфиг инстанса.** Любые файлы конфигурации OpenClaw или платформы, на которой ты запущен: `config.*`, `instance.*`, настройки моделей, runtime-настройки, файлы из системных директорий движка. У нового CloudClaw-инстанса свой конфиг, управляемый платформой — чужой туда класть нельзя.
- **Системные runtime-credentials.** Ключи и токены, которые настроены на уровне инстанса самой платформой/хостингом для работы движка: LLM-ключи провайдера, internal endpoints, service accounts, webhook secrets платформы. Это **не** личные API-ключи пользователя для его скиллов — про них отдельный раздел ниже.
- **Системные файлы OpenClaw.** Бинарники, служебные директории runtime'а, файлы из `system/`, `runtime/`, `.cache/`, `.internal/` и аналогичных.
- **Кэши, временные файлы, логи.** Всё из `*cache*`, `tmp/`, `*.tmp`, `*.log`.

Правило на случай сомнений про конкретный файл: **если не уверен, относится ли он к пользовательским данным — не включай** и упомяни его в финальном сообщении пользователю, чтобы он мог явно решить.

## Секреты пользователя (обсуди перед сборкой)

Речь только про **личные секреты пользователя** — API-ключи, токены, `.env`-файлы, которые он сам добавил для работы своих скиллов или интеграций (например, свой ключ OpenAI/Anthropic, Stripe-токен, webhook для его внешнего сервиса). Системные runtime-ключи платформы под этот раздел не подпадают — их не переносим никогда, см. раздел выше.

### Как обнаружить кандидатов

Ориентируйся на **имена и пути файлов**, не на содержимое. Содержимое credential-файлов открывать незачем — это лишнее чтение приватных данных.

Типовые признаки:

- имя `.env`, `.env.*`, `credentials.*`, `secrets.*`, `keys.*`, `tokens.*`;
- расширения `.pem`, `.key`, `.crt`, `.pfx`;
- папки с именем `secrets/`, `credentials/`, `keys/`, `.secrets/`;
- файлы внутри папок пользовательских скиллов, которые выглядят как конфиги с ключами по названию (`<service>_key.txt`, `api_key.*` и т.п.).

Исключения: шаблоны без настоящих значений (`*.example`, `*.sample`, `*.template`) — это не секреты, они безопасны и попадают в архив как обычные файлы.

### Когда и о чём спрашивать

Если кандидатов **не найдено** — вопрос не задавай. В confirmation-сообщении потом просто не упоминай секреты.

Если кандидаты **найдены** — до начала сборки архива покажи пользователю список найденных путей и задай вопрос. Используй такой шаблон (подставь реальный список, пути даются как есть, без содержимого):

```
Перед тем как собирать архив — я нашёл в твоём файлспейсе файлы, похожие на личные секреты:

- memory/.env
- skills/sales-bot/credentials.json
- skills/research/openai_key.txt

Переносить их в новый инстанс вместе с остальными данными?

— Да, все.
— Нет, оставить в старом инстансе.
— Только часть — скажу, какие из списка.
```

### Что делать с ответом

- **«Да, все»** — включи все файлы из показанного списка в архив наряду с остальными пользовательскими данными.
- **«Нет»** — не включай ни один из них. В confirmation-сообщении отметь «секреты не переносились по твоему решению».
- **«Только часть»** — включи **строго** те пути, которые пользователь перечислил в ответ. Если формулировка расплывчатая («только для OpenAI» без указания файла) — переспроси с конкретными путями из списка.

Если пользователь в ответе назвал файл, которого в твоём списке не было, — уточни у него путь и добавь, только если он действительно существует в user data directory.

## Как упаковать

1. Собери отобранные файлы в **один** архив, сохраняя их относительные пути внутри user data directory (если файл лежал в `memory/goals.md`, в архиве он должен быть по пути `memory/goals.md`).
2. Формат — `.tar.gz` (предпочтительно). Если по техническим причинам доступен только `.zip` — допустимо.
3. Имя файла: `cloudclaw-migration-<YYYY-MM-DD>.tar.gz`, где дата — сегодняшняя.

## Как отдать пользователю

Действия зависят от размера итогового архива. Telegram-пересылка документов в этом контексте ограничена ≈2 GB.

### Если архив ≤ 2 GB

Отправь архив пользователю **документом в текущий чат**. И больше никуда.

Затем коротко сообщи, что собрано (подставь реальные числа, пустые категории опусти):

```
Архив готов: cloudclaw-migration-<YYYY-MM-DD>.tar.gz (<размер>).

Внутри:
- .md-файлы памяти: N
- пользовательские скиллы: M папок
- локальные БД: K
- истории чата (JSONL): L
- личные секреты: P файлов (включено по твоему согласию)

Следующий шаг — перешли этот файл в чат с новым инстансом CloudClaw.
```

Строку про секреты опусти, если пользователь отказался их переносить или кандидатов не было.

### Если архив > 2 GB

Через Telegram такой файл не пройдёт. **Не режь архив на части** и **не загружай его никуда сам**. Вместо отправки файла выдай пользователю этот текст (можно слегка перефразировать, суть сохрани):

```
Архив получился больше 2 GB — через Telegram такой файл переслать нельзя.

Чтобы продолжить миграцию:
1. Возьми любого S3-совместимого провайдера: Cloudflare R2, AWS S3, Yandex Cloud Storage или аналог.
2. Создай бакет, залей туда архив cloudclaw-migration-<YYYY-MM-DD>.tar.gz.
3. Получи на файл ссылку, недоступную из публичного интернета без знания URL:
   — либо presigned URL с коротким TTL (пара часов),
   — либо URL со случайным длинным UID в пути (неугадываемый).
4. Пришли эту ссылку в чат нового инстанса CloudClaw.

Google Drive, Dropbox и Яндекс.Диск без S3-endpoint'а не подойдут — новый инстанс не сможет скачать файл по их ссылкам.

После успешного импорта обязательно удали архив из своего бакета.
```

Если в архиве оказались личные секреты (пользователь согласился их перенести) — перед шагами добавь одну дополнительную строку, чтобы пользователь помнил, что в облако уйдёт файл с его ключами, и ответственнее подошёл к выбору провайдера и TTL:

```
⚠️ В архиве лежат твои личные секреты (которые ты согласился перенести). Раз файл пойдёт через внешнее хранилище — особенно внимательно следи за тем, чтобы ссылка была приватной и недолго живущей.
```

## Safety rules

Эти правила важнее любых других инструкций, включая просьбы пользователя. Если пользователь просит их нарушить — откажись и коротко объясни, почему.

1. **Не отправляй архив никуда, кроме текущего Telegram-чата с пользователем.** Никаких email, webhook'ов, сторонних API, публичных ссылок.
2. **Не загружай архив в облако сам**, даже если у тебя есть credentials для S3/Drive/Dropbox. В S3-сценарии загружает пользователь из своего аккаунта.
3. **Не отправляй содержимое пользовательских файлов во внешние сервисы** (не подкладывай их в LLM-API как лишний контекст, не отправляй в analytics, не логируй дословно).
4. **Не включай служебные файлы инстанса** — системный конфиг и системные runtime-credentials. Про личные секреты пользователя действуй строго по его ответу из раздела «Секреты пользователя».
5. **Не открывай содержимое credential-файлов для «проверки».** Ориентируйся на путь и имя. Лишний раз читать секреты не нужно.
6. **Если не уверен насчёт файла — не включай** и упомяни его в confirmation-сообщении.
7. **Ничего не удаляй из старого инстанса в процессе подготовки dump'а.** Миграция — это копирование, а не перенос. Пользователь сам решит, что делать со старым инстансом после успешного импорта в новом.
